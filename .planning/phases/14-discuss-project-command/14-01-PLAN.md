Create `commands/branchos:discuss-project.md` with the bookend pattern from Phase 12's research command. The file structure:

**Frontmatter:**
```yaml
---
description: Create a PR-FAQ document through guided interactive discussion
allowed-tools: Read, Glob, Grep, Write, Bash(git *), Bash(npx branchos *), WebSearch, WebFetch, AskUserQuestion
---
```

**Step 1: Parse arguments** -- Parse `$ARGUMENTS`. No flags needed for this command. Treat all text as optional project context.

**Step 2: Frame the discussion (Opening Bookend)** -- Load codebase context by reading `.branchos/shared/codebase/ARCHITECTURE.md` if it exists. Check if `.branchos/shared/PR-FAQ.md` already exists -- if so, inform the user and offer to create a fresh PR-FAQ or refine the existing one. Explain the PR-FAQ format briefly (Amazon-style press release + FAQ). Mention the 8 sections that will be covered: Headline, Subheadline, Problem, Solution, Leadership Quote, Call to Action, Customer FAQ, Internal FAQ.

**Step 3: Interactive PR-FAQ creation (Claude Drives)** -- Walk through sections using AskUserQuestion. Use natural conversation flow, NOT rigid form-filling. Group related sections:

- Start with the vision: "What are you building? Who is it for?" (feeds Headline and Subheadline)
- Explore the problem: "What problem does this solve? Why does it matter?" (feeds Problem)
- Define the solution: "How does your project solve this?" (feeds Solution)
- Capture the voice: "If a leader were endorsing this, what would they say?" (feeds Quote)
- Call to action: "How would someone get started with this?" (feeds Call to Action)
- Customer FAQ: "What would potential users ask about this?" (feeds Customer FAQ)
- Internal FAQ: "What would stakeholders or team members ask?" (feeds Internal FAQ)

Include adaptive questioning guidelines: adapt follow-up questions based on the user's responses. If the user provides rich detail in one area, do not ask redundant questions. If an area needs more depth, probe further. Always include an "Other" option or invitation to redirect.

**Step 4: Save PR-FAQ (Closing Bookend)** -- Before saving, review which sections have been covered. Warn if any of the 8 sections are missing and offer to fill gaps. Compile all sections into a well-formatted PR-FAQ markdown document with the canonical section headings that match `EXPECTED_SECTIONS` patterns:
- `## Headline`
- `## Subheadline`
- `## Problem`
- `## Solution`
- `## Quote`
- `## Call to Action`
- `## Customer FAQ`
- `## Internal FAQ`

Write the compiled document to `./PR-FAQ.md` using the Write tool. Then run `npx branchos ingest-prfaq --force` to validate, store, hash, and commit. Report success with a summary of sections covered.

Include `$ARGUMENTS` at the end of the file (required by all slash commands).
