# Phase 15: GWT Acceptance Criteria - Context

**Gathered:** 2026-03-13
**Status:** Ready for planning

<domain>
## Phase Boundary

Feature files support Given/When/Then structured acceptance criteria with backward-compatible freeform fallback. Parser handles GWT blocks in feature file body. Context packets render parsed GWT as checklists. plan-roadmap generates GWT-formatted criteria for new features. No Cucumber/step definitions — GWT is for human-readable criteria only.

</domain>

<decisions>
## Implementation Decisions

### GWT Syntax Rules
- All three keywords required: Given, When, and Then must all be present for a valid GWT block
- And keyword continues the preceding keyword type (standard Gherkin behavior)
- Unnamed blocks — no scenario labels, just Given/When/Then sequences
- Malformed GWT treated as freeform text — graceful fallback, no errors or warnings

### Feature File Format
- GWT blocks live under a dedicated `## Acceptance Criteria` section in the feature body
- Multiple GWT blocks separated by numbered headings: `### AC-1`, `### AC-2`, etc. (short ID only, no titles)
- Mixed mode supported: the AC section can contain both GWT blocks (under ### AC-N) and plain checklist items (`- [ ]` lines)
- Parser looks specifically for the `## Acceptance Criteria` heading to find GWT content

### Context Packet Display
- Parsed GWT rendered as checklist format: `- [ ] **AC-1**` with indented Given/When/Then lines
- Freeform criteria rendered as simple checkboxes in the same unified checklist
- GWT criteria included in context packets for all workflow steps (discuss, plan, execute)
- Transformation happens in `formatFeatureContext` — parsed structured output replaces raw body pass-through

### Roadmap Generation
- plan-roadmap generates 2-4 GWT blocks per feature (covers happy path + key edge cases)
- User-facing language in generated GWT ("Given a developer with a linked feature" not "Given meta.featureId is non-null")
- New features generated by plan-roadmap use exclusively GWT format (freeform fallback for legacy/manual only)
- GWT format specified in /branchos:plan-roadmap slash command instructions (not a separate template file)

### Claude's Discretion
- Exact parser implementation approach (regex, line-by-line, etc.)
- How to structure the parsed GWT type definitions
- Error handling for edge cases not covered above
- Exact checklist formatting details in context packets

</decisions>

<specifics>
## Specific Ideas

- GWT checklist format in context packets should match what Phase 18 (create-pr) will use in PR bodies — design once, reuse
- The `## Acceptance Criteria` section heading is the parser's entry point — everything above it is feature description
- AC-N numbering is positional within the feature file, not globally unique across features

</specifics>
