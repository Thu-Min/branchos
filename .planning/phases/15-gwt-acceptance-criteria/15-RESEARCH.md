# Phase 15: GWT Acceptance Criteria - Research

**Researched:** 2026-03-13
**Domain:** Markdown parsing, structured text extraction, context packet formatting
**Confidence:** HIGH

## Summary

Phase 15 adds Given/When/Then (GWT) acceptance criteria parsing to BranchOS feature files. This is a self-contained text parsing problem with no external dependencies required. The project already has an established pattern for hand-rolled parsers (the frontmatter parser in `src/roadmap/frontmatter.ts`), and this phase follows the same approach for body content parsing.

The implementation touches four areas: (1) a pure GWT parser function that extracts structured criteria from the `## Acceptance Criteria` section of feature file bodies, (2) type definitions for the parsed output, (3) formatting logic in `formatFeatureContext` to render parsed GWT as checklists in context packets, and (4) updating the `branchos:plan-roadmap` slash command instructions to generate GWT-formatted criteria.

**Primary recommendation:** Build a line-by-line parser (not regex-based) that mirrors the frontmatter parser's approach -- pure function, no I/O, graceful fallback to freeform text for malformed input.

### Pattern 1: Line-by-Line Parser (Recommended)
**What:** Process body text line by line, tracking parser state (current section, current AC block, current keyword type for And-continuation)
**When to use:** Parsing structured text with section headers and keyword-prefixed lines
**Example:**
```typescript
// New file: src/roadmap/gwt-parser.ts

export interface GwtStep {
  keyword: 'Given' | 'When' | 'Then';
  text: string;
}

export interface GwtBlock {
  id: string;         // "AC-1", "AC-2", etc.
  steps: GwtStep[];
}

export interface FreeformCriterion {
  text: string;       // The checklist item text (without "- [ ] " prefix)
}

export interface ParsedAcceptanceCriteria {
  gwtBlocks: GwtBlock[];
  freeformItems: FreeformCriterion[];
}

export function parseAcceptanceCriteria(body: string): ParsedAcceptanceCriteria {
  // 1. Find "## Acceptance Criteria" section
  // 2. For each "### AC-N" sub-heading, extract GWT lines
  // 3. Validate: all three keywords (Given, When, Then) must appear
  // 4. If invalid, treat block content as freeform
  // 5. Collect loose "- [ ]" lines as freeform items
  // ...
}
```

### Pattern 2: formatFeatureContext Enhancement

**When to use:** When generating context packet output
**Example:**

// In src/cli/context.ts, updated formatFeatureContext:
function formatFeatureContext(feature: Feature): string {
  const header = [ /* existing table */ ].join('\n');

  const parsed = parseAcceptanceCriteria(feature.body);
  const criteriaSection = formatGwtChecklist(parsed);

### Pattern 3: Body Splitting (Description vs AC)
**What:** The feature body contains both description text (above `## Acceptance Criteria`) and the AC section. The parser must preserve the description while only transforming the AC section.
**When to use:** When rendering feature context -- description passes through untouched, AC section gets parsed and reformatted.
**Example:**
```typescript
// Split body at "## Acceptance Criteria" heading
function splitBody(body: string): { description: string; acSection: string | null } {
  const acIndex = body.indexOf('## Acceptance Criteria');
  if (acIndex === -1) return { description: body, acSection: null };
  return {
    description: body.slice(0, acIndex).trim(),
    acSection: body.slice(acIndex).trim(),
  };
}
```

### Anti-Patterns to Avoid
- **Modifying the Feature type to store parsed GWT:** The `Feature.body` remains a raw string. Parsing happens at render time in `formatFeatureContext`. This avoids schema changes and keeps the read/write path simple.
- **Throwing errors on malformed GWT:** The decision explicitly says malformed GWT is treated as freeform text. Never throw, never warn.
- **Over-engineering the parser for Cucumber compatibility:** This is NOT a Gherkin parser. No Scenario/Feature/Background keywords. No step definitions. Just Given/When/Then/And extraction.

## Common Pitfalls

### Pitfall 1: And-Keyword Ambiguity
**What goes wrong:** `And` lines without a preceding Given/When/Then keyword have no parent to attach to.
**Why it happens:** A feature file might have `And` as the first line in a block, or after a blank line that "resets" context.
**How to avoid:** Track `lastKeyword` state. If `And` appears with no preceding keyword, treat the entire block as freeform (per the malformed-GWT fallback rule).
**Warning signs:** Tests with `And` at the start of a block should verify freeform fallback.

### Pitfall 2: Section Heading Confusion
**What goes wrong:** Parser matches `## Acceptance Criteria` inside code blocks, or matches `### AC-1` headings outside the AC section.
**Why it happens:** Naive line matching without section awareness.
**How to avoid:** Only parse content after finding the `## Acceptance Criteria` heading. Stop at the next `## ` heading (if any). For MVP, code blocks inside AC sections are unlikely, but a simple "inside code fence" flag is cheap insurance.
**Warning signs:** Feature files with `##` headings after the AC section.

### Pitfall 3: Forgetting to Preserve Non-AC Body Content
**What goes wrong:** The feature description (text before `## Acceptance Criteria`) gets lost when reformatting.
**Why it happens:** Treating the entire body as AC content.
**How to avoid:** Split body into description + AC section. Only transform AC section. Recombine in output.
**Warning signs:** Tests should verify that feature descriptions appear in context packet output.

### Pitfall 4: GWT Validation -- Missing Keywords
**What goes wrong:** A block with only `Given` and `When` (no `Then`) gets treated as valid GWT.
**Why it happens:** Not checking for all three required keywords.
**How to avoid:** After parsing a block, verify it contains at least one Given, one When, and one Then step. If not, demote to freeform.
**Warning signs:** Unit tests for partial GWT blocks.

### Pitfall 5: Checklist Format Inconsistency with Phase 18
**What goes wrong:** Phase 18 (create-pr) needs to render GWT checklists in PR bodies. If the format is defined differently, it needs to be refactored.
**Why it happens:** Format defined in one place, consumed in another.
**How to avoid:** Extract the GWT-to-checklist formatting into a reusable function (e.g., `formatGwtChecklist` in `gwt-parser.ts` alongside the parser). Phase 18 can import and reuse it.
**Warning signs:** The CONTEXT.md explicitly calls this out: "GWT checklist format in context packets should match what Phase 18 (create-pr) will use in PR bodies -- design once, reuse."

## Open Questions

1. **Checklist rendering: Should And lines show as "And" or expand to parent keyword?**
   - What we know: Standard Gherkin shows "And" as a keyword. The CONTEXT.md shows "And" in the rendered example.
   - Recommendation: Render as `And` (keeps it concise and matches Gherkin convention). The parsed type stores the resolved keyword, but formatting outputs "And" for lines that were originally "And".

2. **Feature description text before `## Acceptance Criteria` -- include in context or not?**
   - What we know: Current `formatFeatureContext` passes the entire body. The CONTEXT.md says "parsed structured output replaces raw body pass-through."
   - Recommendation: Include feature description (pre-AC text) in context output, then render parsed AC below it. The "replaces raw body" means the AC section is reformatted, not that the description is dropped.
